# WeClawBot 官网真机交互体验需求

本文档描述 `weclawbot.link` 官网首页中的“像操作一台真机一样试用微笺屏”体验。它不是普通产品介绍页，也不是静态效果图，而是一个可扫码、可收微信消息、可实时改变屏幕内容的网页真机模拟器。

## 背景

WeClawBot 的核心产品体验是：用户用微信扫码连接一块 4.2 寸墨水屏，随后通过微信发送文字、图片或文件，云端整理成适合常驻桌面的留言、提醒、清单、照片相框等内容，再下发到屏幕显示。

真实硬件是 Waveshare ESP32-S3-RLCD-4.2。官网需要让首次访问者在没有硬件的情况下，也能通过自己的微信体验这条链路：

1. 在网页中看到一台可旋转、可缩放的 WeClawBot 真机。
2. 用自己的微信扫描网页二维码，建立一次临时体验连接。
3. 在微信里发送留言、提醒、清单或照片。
4. 网页中的 4.2 寸屏幕像真实设备一样刷新显示。
5. 微信端收到与屏幕一致的结果反馈图。

## 产品定位

官网体验应该传达的不是“网页里有一个屏幕预览”，而是“我正在远程操作一台真实的微笺屏”。

体验关键词：

- 一对一，无干扰。
- 微信即输入端。
- 墨水屏是固定场景里的常驻留言界面。
- 留言、清单、照片都应该有文化感、亲密感和生活感。
- 用户不需要理解云函数、固件、配置流程，第一次进入官网即可试用。

## 范围

本需求只覆盖官网交互体验项目，不覆盖 ESP32 固件实现、云端 curator 的模型策略、真实设备烧写配置页。

官网体验项目需要能接入现有云端整理接口和 ilink 微信连接能力，但自身不公开私有云端实现细节。

## 页面结构

首页首屏建议由两部分组成：

- 左侧或主区域：可交互的 3D WeClawBot 真机模型。
- 右侧或辅助区域：微信扫码连接面板、本地回环测试、操作按钮、运行日志。

页面不应做成传统营销 landing page。用户打开后第一眼就应该看到“正在操作的设备”，并且马上能扫码试用。

## 3D 真机模型要求

### 基本要求

必须使用 Three.js 或同等级 WebGL 技术绘制真机模型，不使用一张产品照片作为底图再贴屏幕。

允许屏幕内容使用动态 CanvasTexture，因为真实显示内容本身就是实时生成的像素图。但屏幕内容必须嵌入设备结构中，不能像一张白纸漂浮或贴在机器前方。

模型至少包含：

- 4.2 寸 400x300 墨水屏面板，比例为 4:3。
- 黑色外框和内凹屏幕结构。
- 灰白墨水屏纸面材质。
- 透明或半透明玻璃/保护膜质感。
- 左侧 PCB 边缘。
- 左侧固定螺柱或白色支撑柱。
- 右侧橙色物理按键。
- 顶部 BOOT/RESET 等小按键的视觉暗示。
- USB-C 插头或接口。
- 背后支架。
- 接地阴影，让设备像放在桌面上。

### 视觉标准

模型应看起来像 Waveshare ESP32-S3-RLCD-4.2 这类裸板墨水屏模块，而不是通用平板、手机、相框或玩具屏幕。

重点比例：

- 屏幕主体接近 4:3。
- 屏幕应占据设备正面大部分区域。
- 黑色边框不能过厚。
- 右侧橙色按键只能是侧边小结构，不能喧宾夺主。
- 左侧 PCB 和螺柱可以作为识别真机的细节。
- 支架应该在斜视角下可见，形成“桌面设备”的真实感。

### 交互要求

用户可以：

- 鼠标或触摸拖拽旋转设备。
- 鼠标滚轮缩放设备。
- 旋转范围要受限，避免穿模或看到明显破绽。
- 首屏默认角度应该像用户斜着看桌面上的真机，而不是正交平面图。

移动端：

- 3D 设备仍然可见。
- 标题和控制面板不能遮挡设备。
- 操作区可以下移，但扫码和本地测试功能必须可用。

## 屏幕显示要求

屏幕显示内容应该与真实固件的视觉方向保持一致：

- 分辨率基准为 400x300。
- 黑白或灰阶墨水屏风格。
- 支持由云端下发像素图页面。
- 支持多页自动翻页。
- 支持无内容时显示桌宠或照片相框。
- 支持处理消息时显示悬浮红方块桌宠的“思考中”状态。

### 内容类型

官网体验至少支持以下屏幕状态：

- 未连接：红方块待机或等待扫码状态。
- 等待扫码：屏幕提示用微信扫码连接。
- 已连接无内容：红方块桌宠或默认屏保。
- 正在处理：红方块悬浮/跳动，显示思考状态。
- 文字留言：按云端返回的屏幕像素页显示。
- 清单/待办：可跨页显示，自动翻页。
- 照片：生活照或宠物照可成为黑白照片相框。
- 错误：用温和、可理解的方式提示用户，而不是暴露技术错误。

### 像素图优先

官网屏幕应优先渲染云端返回的 `screen_pages` 像素图，确保网页预览、微信回图和真实设备显示尽量一致。

如果云端只返回文本，前端可以用降级排版渲染，但这只能作为 fallback，不应成为主路径。

## 微信扫码体验

### 会话创建

用户点击“创建扫码体验”后：

1. 官网后端创建短生命周期体验会话。
2. 后端向 ilink 请求微信登录二维码。
3. 网页显示二维码。
4. 用户用微信扫码确认。
5. 网页状态变为已连接。

二维码只用于当前网页体验会话，不应展示长期 token 或私有服务凭据。

### 消息处理链路

微信消息进入后，官网体验应按以下链路处理：

1. 接收微信消息。
2. 转换成与真实设备一致的 curator 输入事件。
3. 携带当前屏幕文本上下文和当前像素页摘要。
4. 调用云端 curator。
5. 获取新的 canonical text、screen pages、reply text、preview image。
6. 更新网页 3D 真机屏幕。
7. 通过微信返回处理结果和屏幕效果图。

### 微信端反馈

无论用户发送纯文本、图片还是可处理文件，只要内容成功上屏，微信端都应该收到反馈。

反馈应包含：

- 简短说明：已覆盖到屏幕、已加入清单、已替换照片等。
- 若有多页，说明总页数和自动翻页。
- 屏幕最终效果图。

不应给普通用户返回预览 URL、内部 CDN URL、云函数地址、日志地址等技术信息。

## 本地回环测试

为了方便开发和演示，官网体验页应保留本地回环测试输入框。

本地测试不依赖微信扫码，可以模拟一条微信文字消息进入 curator，并更新 3D 屏幕。

本地测试目的：

- 快速验证 3D 屏幕更新。
- 快速验证云端 curator 输出。
- 方便没有微信连接时演示基础体验。

本地测试不能替代微信扫码真实链路。

## 控制按钮

体验页可提供模拟物理按键：

- 左键短按：上一页。
- 右键短按：下一页。
- 左键长按：清除文字留言，保留照片或微信登录态。
- 右键长按：全清，清除图片、文字、微信登录和 Agent 配对；官方模式返回
  微信扫码等待，BYOA 模式返回六位配对码。

这些按钮用于网页体验中的模拟，不代表必须暴露在最终用户主路径中。

## 隐私与安全边界

官网前端不得暴露：

- ilink bot token。
- cursor。
- 微信私有登录态。
- 云端整理服务的内部密钥。
- 腾讯云函数真实内部地址。
- 用户上传文件的内部对象存储地址。

前端可以知道：

- 当前体验会话 ID。
- 二维码内容。
- 屏幕显示状态。
- 日志中的非敏感状态事件。

官网后端应该只保存短生命周期会话。体验会话过期后应清理微信连接和内存状态。

## 日志要求

开发环境需要有可见日志，方便排查“微信没反应”“屏幕没更新”“云端没返回”等问题。

日志至少记录：

- 会话创建。
- 二维码获取。
- 扫码状态变化。
- 微信消息类型。
- curator 请求开始/结束。
- 屏幕页数。
- 微信回图发送结果。
- 错误摘要。

日志不能记录完整 access token、用户隐私文件 URL、长期凭证。

## 多用户体验

官网体验需要支持多个网页访客同时打开。

每个访客应该拥有独立体验会话：

- 独立二维码。
- 独立微信连接状态。
- 独立屏幕内容。
- 独立日志。

当前阶段不要求一个微信同时绑定多台屏。微信官方规则下，一个个人微信同一时间只连接一个 bot，因此官网体验可按“一次扫码连接一台虚拟屏”设计。

## 兼容真实设备体验

官网体验不是固件配置页，但它应该尽量复用真实设备的数据结构：

- curator 输入事件结构一致。
- `screen_render` / `screen_pages` 输出结构一致。
- 多页自动翻页逻辑一致。
- 清屏、换页、照片相框等概念一致。

这样官网体验可以成为真实设备链路的可视化调试入口，而不是另一套孤立 demo。

## 非目标

当前阶段不要求：

- 在官网直接烧写固件。
- 在体验页公开云端 curator 实现。
- 在浏览器里运行完整 AI agent。
- 在 Three.js 中执行动态 LVGL 二进制。
- 完整模拟 ESP32 的 Wi-Fi 配网、NVS、断电恢复。
- 追求工业级 CAD 精度。

## 验收标准

### 视觉验收

- 打开官网后，用户第一眼能看出这是一块 4.2 寸墨水屏设备。
- 屏幕内容看起来嵌在设备里，不是贴在前面的平面图。
- 设备拖拽旋转时，支架、厚度、侧边按键等结构成立。
- 默认视角像真实桌面上斜着看一台设备。
- 移动端不出现文字溢出、控件遮挡或横向滚动。

### 功能验收

- 能创建二维码。
- 能扫码连接微信。
- 微信发送文本后网页屏幕更新。
- 微信发送图片后可变成黑白照片相框或给出合理反馈。
- 成功上屏后微信端返回屏幕效果图。
- 多页内容自动翻页。
- 本地回环测试可用。
- 模拟左右键可翻页和清除内容。

### 安全验收

- 浏览器源码中不能看到 bot token。
- 微信回包中不能出现预览 URL 或内部文件 URL。
- 日志不泄露长期凭证。
- 体验会话过期后不可继续使用旧二维码控制屏幕。

## 推荐实现结构

建议项目结构：

```text
site-experience/
  index.html
  server.mjs
  src/
    main.js
    device-model.js
    screen-renderer.js
    style.css
  public/
  README.md
```

建议职责：

- `device-model.js`：只负责 Three.js 真机建模。
- `screen-renderer.js`：只负责把云端 screen pages 渲染成 CanvasTexture。
- `main.js`：负责状态、事件、微信会话、用户交互。
- `server.mjs`：负责二维码、SSE、ilink 回调、curator 转发。

## 设计判断

屏幕内容使用 CanvasTexture 是合理的，因为真实设备收到的也是渲染后的屏幕内容。但官网体验必须避免“贴图感”：需要靠设备几何结构、内凹层次、灰白屏面、玻璃层和正确比例，让用户感知它是一块真实屏幕。

这也是后续 WeClawBot Layout VM 的入口：今天下发的是像素图，未来可以演进为更动态的屏幕布局描述或可执行显示逻辑，但官网体验的第一原则始终是让用户相信自己正在操作一台真实微笺屏。
