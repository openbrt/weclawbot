# WeClawBot Layout VM

本文记录一个后续方向：云端不向 ESP32 下发动态 LVGL 原生二进制，而是下发受限、可校验、可回滚的声明式布局程序。

## 当前阶段

先实现“像素内容区显示”：

- 固件继续绘制固定上横杆、下横杆、页码和底部操作提示；
- 云端生成中间内容区的 1-bit 像素图；
- 设备用 LVGL image 把像素图贴到内容区；
- 设备将当前最多三页云端像素图持久化到 `storage` 分区，重启或刷新应用固件后继续使用同一布局；
- 若云端像素图不可用或尺寸不合法，设备回退到本地文字排版。
- 云端使用同一份布局计划生成微信预览和设备内容区像素图。
- 云端把规范文本和像素页作为同一个 `screen_state` 下发，并用 `revision` / `render_id` 绑定；设备在下一条消息中回传这份规范文本作为当前屏幕上下文。
- `screen_state` 作为单个原子状态包持久化；若更新中途掉电，设备恢复完整旧状态或完整新状态，不会出现新文本配旧像素图。
- 下一轮整理器基于当前 `canonical_text` 和新来内容决定 `update_note`、`merge_note`、`replace_note` 或保持不变，并始终返回完整的新屏幕状态，而不是只返回增量。
- 布局计划在不超过三页的约束内逐像素寻找最大可用字号，并让每页内容块在内容区内视觉居中。自动翻页是可用的显示空间，不为了强塞进一页而缩小文字。
- 字体按内容语义选择：家庭留言、书信和读书笔记使用文楷；提醒、清单和验证码使用黑体；诗词、格言和祝福使用书法体。
- 云端版式模式按语义分流：普通留言走 `adaptive_fill`；短诗、格言、祝福等走 `poetry_scroll`，使用书法字体、诗笺题签、四角笺框和黑白印章来形成更强的文化气质。
- 多屏隔离以微信连接身份为主键：固件把 ilink 登录返回的 `ilink_bot_id` 包装成与 PWA 版一致的 `wechat_id = "u_" + ilink_bot_id` 上报。没有微信连接就无法修改屏幕，因此当前阶段不引入额外账号体系或独立 `device_id`。
- 自动翻页时设备输出 `ui/auto_page_turn` 串口事件，便于无需相机完成分页联调。

当前内容区约定：

- 屏幕：`400 x 300`
- 内容区：`x=16, y=42, w=368, h=206`
- 页数：最多 3 页，设备自动翻页
- 当前云端字体：霞鹜文楷、Noto Sans CJK SC、Ma Shan Zheng

照片相框是独立于微笺的 idle surface：

- `set_idle_photo` 下发 `viewport=fullscreen` 的 `400 x 300` `mono1` 像素页；
- 设备把相框存到 SPIFFS `/storage/idle_photo.bin`，不会随 `/clear` 或 `CLEAR_NOTES` 删除；
- 当前微笺优先显示；当微笺为空时，显示用户照片相框；
- 如果没有用户照片，才显示系统自带红方块桌宠；
- 这条路径为后续 `WeClawBot Layout VM` 的全屏 `image_mono1` 指令保留了语义位置。

## 后续方向

WeClawBot Layout VM 可以成为云端整理器和设备显示之间的中间层。它不是动态原生代码执行，而是一套小型声明式指令集，例如：

- `text`：在指定区域绘制文字；
- `image_mono1`：绘制 1-bit 图像块；
- `line` / `rect`：绘制分割线和简单边框；
- `pet_state`：请求红方块桌宠进入空闲、思考、提醒等状态；
- `page`：定义多页内容和自动翻页策略。

设备端只解释白名单指令，并限制内存、页数、坐标、字体、动画和执行时间。云端模型可以提出布局计划，但最终必须通过 schema 校验后才能下发。

这个方向的目标是保留“云端决定屏幕格式化/显示框架”的灵活性，同时避免远程执行动态 LVGL 二进制带来的崩溃、安全和升级风险。
